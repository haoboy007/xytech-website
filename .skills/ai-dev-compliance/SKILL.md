# AI应用开发合规与侵权审查技能

## 一、能力概述

本技能为AI vibe coding开发者提供**系统化的应用上线前合规审查框架**，覆盖知识产权、开源许可、数据隐私、内容合规、平台规则、AI专项法规六大维度。核心目标是在应用上线前识别并消除法律风险，避免因合规问题导致应用下架、诉讼或行政处罚。

**核心价值：**
- 将分散的法律法规要求转化为可执行的检查清单
- 针对一人开发者的资源限制，提供低成本、可落地的整改方案
- 区分"必须立即修复"与"建议后续完善"的风险等级
- 输出可直接使用的合规文档模板（隐私政策、用户协议、版权声明等）

---

## 二、触发后立即执行的诊断流程

当用户提出合规审查需求时，不要急于输出结论。先通过以下问题快速定位项目背景和风险敞口：

### 2.1 项目信息收集（必问）

向用户确认以下信息（可用对话形式逐步确认）：

| 维度 | 关键问题 | 为什么重要 |
|------|---------|-----------|
| **应用类型** | 这是什么类型的应用？（工具/社交/电商/内容/金融/教育/医疗/游戏） | 不同行业触发不同监管要求 |
| **目标市场** | 主要面向中国大陆、海外（哪些国家/地区）、还是全球？ | 决定适用哪些法律体系 |
| **技术栈** | 使用了哪些开源框架/库？AI模型是自研、开源、还是调用API？ | 直接影响许可证和API条款合规 |
| **数据收集** | 是否收集用户个人信息？收集哪些类型？是否涉及生物识别、位置、通讯录？ | 触发数据保护法规 |
| **AI使用方式** | AI用于生成内容（文案/图片/视频/代码）、推荐算法、还是其他？输出是否面向公众？ | 触发深度合成、算法推荐等专项规定 |
| **素材来源** | 应用中的图片、字体、音乐、代码片段来自哪里？（AI生成/开源/自研/第三方购买/不确定） | 直接影响侵权风险评估 |
| **商业模式** | 是否有付费、订阅、广告、电商交易？涉及用户资金吗？ | 触发消费者权益、金融合规 |
| **用户群体** | 是否面向未成年人（尤其是14岁以下）？ | 触发儿童个人信息保护、防沉迷等要求 |

### 2.2 快速风险信号识别

在对话中主动扫描以下高风险信号，一旦发现立即标记为"紧急关注"：

- 🔴 **使用GPT/Claude等API输出直接作为商用产品内容**（可能违反API使用条款）
- 🔴 **应用图标/UI直接复制知名品牌设计风格**（商标/不正当竞争风险）
- 🔴 **爬取第三方数据作为核心功能**（数据侵权、爬取合规风险）
- 🔴 **收集用户数据但未提供隐私政策**（行政处罚风险，最高可达年营业额5%）
- 🔴 **使用GPL许可证的库但未开源自身代码**（许可证违约风险）
- 🔴 **AI生成内容涉及真人形象/声音**（肖像权、深度合成规定风险）
- 🔴 **应用涉及医疗/金融建议但未获得资质**（非法经营风险）

## 三、六大合规审查维度

基于收集到的项目信息，按以下六个维度进行系统性审查。每个维度包含：**检查清单 → 常见陷阱 → 整改建议 → 参考文档索引**。

### 维度1：知识产权侵权审查

**核心关切：** 代码、图片、字体、音乐、文案、商标、专利是否侵犯他人权利。

#### 检查清单

- [ ] **代码原创性**：核心代码是否为原创，或基于MIT/Apache/BSD等宽松许可证？
- [ ] **代码片段来源**：从Stack Overflow、GitHub、教程复制粘贴的代码段是否超过"合理使用"阈值？
- [ ] **图片素材**：应用内所有图片（图标、背景、插图、用户头像默认图）的来源是否合法？
- [ ] **字体使用**：使用的字体是否为免费商用字体（如思源黑体、阿里巴巴普惠体）或已购买授权？
- [ ] **音乐/音效**：背景音乐、提示音是否为免版税或已获授权？
- [ ] **文案内容**：应用内文案、AI生成的商品描述/文章是否抄袭或过度模仿他人作品？
- [ ] **商标冲突**：应用名称、Logo是否与已注册商标（尤其是同类别）存在冲突？
- [ ] **专利风险**：核心功能是否可能落入已知软件专利的权利要求范围？

#### 常见陷阱（一人开发者特别易犯）

| 陷阱 | 说明 | 典型案例 |
|------|------|---------|
| "网上找的图随便用" | 搜索引擎下载的图片99%受版权保护 | 使用Pinterest/百度图片作为应用背景 |
| "系统默认字体肯定能用" | 系统字体仅用于显示，嵌入分发需授权 | 将微软雅黑嵌入Web字体 |
| "改一改就不算抄" | 抄袭认定采用"实质性相似"标准 | 复制竞品UI布局只改颜色 |
| "开源=免费随便用" | 开源≠放弃版权，许可证有约束 | 忽略GPL的"传染性" |
| "AI生成的就是我的" | AI生成内容的版权归属存在法律争议 | 用Midjourney生成的图申请商标被拒 |

#### 整改建议分级

- **立即修复（P0）**：移除或替换所有来源不明的图片/字体；停用可能商标冲突的名称
- **短期内修复（P1）**：为所有开源依赖添加LICENSE文件引用；对AI生成内容添加免责声明
- **建议完善（P2）**：建立素材来源台账；为核心功能做专利检索（FTO分析）

需要深入了解时，读取 `references/ip-infringement-guide.md`。

### 维度2：开源许可证合规

**核心关切：** 使用的开源库、框架、工具是否符合其许可证要求，不同许可证之间是否兼容。

#### 许可证兼容性速查

| 许可证 | 商用 | 修改 | 分发 | 闭源 | 专利授权 | 传染性 | 典型库 |
|--------|------|------|------|------|---------|--------|--------|
| MIT | ✅ | ✅ | ✅ | ✅ | 无 | 无 | React, Vue, lodash |
| Apache-2.0 | ✅ | ✅ | ✅ | ✅ | ✅ | 无 | TensorFlow, Kubernetes |
| BSD-3 | ✅ | ✅ | ✅ | ✅ | 无 | 无 | Flask, NumPy |
| LGPL | ✅ | ✅ | ✅ | 可链接闭源 | 无 | 仅修改部分 | Qt, FFmpeg |
| GPL-2/3 | ✅ | ✅ | ✅ | ❌必须开源 | 无 | 强 | Linux, MySQL, WordPress |
| AGPL | ✅ | ✅ | ✅ | ❌必须开源 | 无 | 极强（网络服务也触发） | MongoDB(旧), Grafana(旧) |
| SSPL | ⚠️ | ⚠️ | ⚠️ | 通常不可 | 无 | 极强 | MongoDB(新), Elasticsearch |
| 闭源/自定义 | 看条款 | 看条款 | 看条款 | 看条款 | 看条款 | 看条款 | 各种SDK |

#### 检查清单

- [ ] **依赖清单**：能否列出所有直接和间接依赖及其许可证？
- [ ] **许可证文件**：每个依赖的LICENSE文件是否随应用分发？
- [ ] **版权声明**：是否在应用中保留了所有必要的版权声明？
- [ ] **传染性分析**：是否使用了GPL/AGPL的库但未计划开源自身代码？
- [ ] **修改披露**：是否修改了LGPL库且未提供修改后的源代码？
- [ ] **专利条款**：Apache-2.0等含专利授权的许可证，是否理解其"反报复"条款？
- [ ] **云服务条款**：使用Firebase、Supabase、Vercel等云服务的条款是否允许当前商业模式？

#### 典型冲突场景

```
你的闭源应用
├── MIT库（安全）
├── Apache-2.0库（安全）
├── LGPL库 → 动态链接/未修改（安全）
├── GPL库 → 直接调用/静态链接（危险！触发开源义务）
└── AGPL库 → 通过网络提供服务（极危险！整个应用需开源）
```

### 维度3：数据隐私与保护合规

**核心关切：** 收集、处理、存储、传输用户个人信息的合法性。

#### 中国法合规（如目标市场含中国大陆）

| 法规 | 核心要求 | 违规后果 |
|------|---------|---------|
| 《个人信息保护法》 | 告知-同意、最小必要、目的明确、数据安全 | 最高5000万元或年营业额5% |
| 《数据安全法》 | 数据分类分级、重要数据识别、安全风险评估 | 最高1000万元 |
| 《网络安全法》 | 网络安全等级保护、数据本地化、出境安全评估 | 最高100万元 |
| 《儿童个人信息网络保护规定》 | 14岁以下需监护人同意、专人负责、最小收集 | 行政处罚 |
| 《App违法违规收集使用个人信息认定方法》 | 无隐私政策、超范围收集、强制索取权限等六类违规 | 通报、下架 |

#### 国际法合规（如目标市场含海外）

| 法规 | 适用条件 | 核心要求 |
|------|---------|---------|
| GDPR（欧盟） | 向欧盟居民提供服务 | 合法基础、数据主体权利、DPO、跨境传输机制 |
| CCPA/CPRA（美国加州） | 年处理≥10万加州居民数据 |  opt-out权利、数据披露、敏感数据限制 |
| PIPEDA（加拿大） | 处理加拿大居民数据 | 知情同意、合理目的、安全保障 |
| LGPD（巴西） | 处理巴西居民数据 | 类似GDPR，需法律基础+DPO |

#### 检查清单

- [ ] **隐私政策**：是否已制定并发布隐私政策？是否以显著方式告知用户？
- [ ] **同意机制**：是否在收集前获得用户同意？同意是否为"单独同意"（敏感信息）？
- [ ] **最小必要**：收集的数据类型和范围是否与功能直接相关？
- [ ] **权限对应**：申请的每一项系统权限是否有对应功能？（Android/iOS权限审查）
- [ ] **数据存储**：用户数据存储在何处？是否加密（传输中+静止状态）？
- [ ] **数据跨境**：是否涉及数据跨境传输？是否完成安全评估/标准合同/认证？
- [ ] **数据留存**：是否有明确的数据保留期限和删除机制？
- [ ] **第三方共享**：是否与第三方共享数据？是否有数据处理协议（DPA）？
- [ ] **数据主体权利**：是否提供查询、更正、删除、导出个人数据的功能？
- [ ] **未成年人保护**：是否识别14岁以下用户？是否有监护人同意机制？

#### 参考文档

需要深入了解时，读取 `references/data-privacy-laws.md`。

### 维度4：AI生成内容专项合规

**核心关切：** 使用AI生成内容时的特殊法律风险，尤其是中国大陆的专项监管。

#### 中国大陆核心法规

| 法规 | 适用范围 | 关键义务 |
|------|---------|---------|
| 《生成式人工智能服务管理暂行办法》 | 面向境内公众提供的生成式AI服务 | 算法备案、内容标识、训练数据合法、用户协议 |
| 《互联网信息服务深度合成管理规定》 | 深度合成服务（换脸、语音合成、AI生成内容） | 显著标识、不得用于虚假信息、安全评估 |
| 《算法推荐管理规定》 | 具有推荐/排序/选择功能的算法 | 算法备案、用户选择权、不得诱导沉迷 |
| 《科技伦理审查办法（试行）》 | 涉及伦理风险的科技活动 | 伦理审查、知情同意、风险受益评估 |

#### 检查清单

- [ ] **算法备案**：如果面向中国公众提供生成式AI服务，是否完成算法备案？
- [ ] **深度合成标识**：AI生成/深度合成的图片/视频/音频是否有显著标识（"此内容系AI生成"）？
- [ ] **虚假信息防范**：是否有机制防止AI被用于生成虚假信息、诈骗内容？
- [ ] **训练数据合法性**：AI模型训练数据是否合法获取？是否有数据清洗和去标识化？
- [ ] **用户协议**：是否明确告知用户AI生成内容的局限性、不准确性、版权归属？
- [ ] **内容审核**：是否建立AI输出内容的审核机制？是否有投诉举报渠道？
- [ ] ** API条款合规**：使用ChatGPT/Claude/Midjourney等API的输出，是否允许用于当前场景？

#### AI API使用条款常见限制（极易被忽视）

| 服务商 | 关键限制 | 常见违规 |
|--------|---------|---------|
| OpenAI | 禁止用于竞争监控、军事、法律自动化建议（部分）；输出不声称来自真人 | 用GPT输出直接作为法律文书 |
| Anthropic | 禁止用于监控/分析个人、自动化决策产生法律后果、制造虚假信息 | 用Claude批量生成虚假评论 |
| Midjourney | 商业用途需付费计划；不声称对生成图像拥有版权 | 免费计划商用 |
| Stable Diffusion | 开源模型较宽松，但特定微调版本可能有额外限制 | 忽略微调版本的附加条款 |

需要深入了解时，读取 `references/ai-specific-regulations.md`。

### 维度5：平台与分发合规

**核心关切：** 应用在不同平台分发时需要遵守的规则，以及云服务/API的使用条款。

#### 主要平台合规要求

| 平台 | 关键合规点 | 常见下架原因 |
|------|-----------|-------------|
| Apple App Store | 隐私营养标签、ATT框架（IDFA授权）、支付内购规则、2.1 App Completeness | 隐私标签与实际收集不符、强制登录、绕过内购 |
| Google Play | 数据安全表单、权限声明、家庭政策、支付政策 | 隐瞒数据收集、过度索权、虚假功能描述 |
| 微信小程序 | 类目资质、用户隐私保护指引、内容安全、ICP备案 | 未配置隐私保护指引、内容违规、无ICP备案 |
| 国内安卓商店 | 工信部APP合规（权限、隐私政策、账号注销）、版号（游戏） | 未提供账号注销、超范围索权 |
| Web/浏览器 | Cookie同意、HTTPS、内容安全政策（CSP）、无障碍 | 未获Cookie同意、混合内容 |

#### 云服务/API条款合规

- [ ] **AWS/Azure/GCP**：免费层/学生套餐是否允许商业用途？流量限制是否满足？
- [ ] **Firebase/Supabase**：是否超出免费额度？数据存储区域是否符合法规？
- [ ] **CDN/图床**：是否允许当前内容类型？是否有流量/带宽限制？
- [ ] **第三方登录**：微信/QQ/微博登录的开发者协议是否允许当前场景？
- [ ] **支付接口**：微信支付/支付宝/Stripe的商户协议和费率是否了解？

#### 参考文档

需要深入了解时，读取 `references/platform-compliance.md`。

### 维度6：行业特殊合规

**核心关切：** 特定行业面临的额外监管要求。

#### 高风险行业清单

| 行业 | 主要法规 | 关键要求 |
|------|---------|---------|
| **金融** | 非银支付、P2P、借贷、虚拟货币相关法规 | 牌照/备案、资金存管、反洗钱、利率合规 |
| **医疗** | 《互联网诊疗管理办法》、医疗器械监管 | 互联网医院资质、医生实名、处方药限制 |
| **教育** | 《校外培训行政处罚暂行办法》 | 禁止学科类培训营利、预付费监管、教师资格 |
| **游戏** | 版号制度、未成年人保护、防沉迷 | 必须取得版号、实名认证、时长/充值限制 |
| **电商** | 《电子商务法》、消费者权益保护 | 营业执照公示、七天无理由、虚假广告禁止 |
| **社交/UGC** | 《网络信息内容生态治理规定》 | 内容审核、用户实名、举报机制、评论区管理 |
| **直播/短视频** | 《网络主播行为规范》 | 实名认证、未成年人保护、打赏限制、内容审核 |

## 四、风险评估与输出框架

### 4.1 风险等级定义

| 等级 | 标识 | 定义 | 处理时效 |
|------|------|------|---------|
| 紧急 | 🔴 P0 | 可能导致应用下架、诉讼、行政处罚或严重经济损失 | 24小时内 |
| 重要 | 🟡 P1 | 可能引发用户投诉、平台警告或中等法律风险 | 1周内 |
| 建议 | 🟢 P2 | 长期合规建设、提升用户信任、预防未来风险 | 1个月内 |

### 4.2 输出报告结构

每次合规审查必须输出结构化的**《合规审查报告》**，包含以下部分：

```markdown
# 【应用名称】合规审查报告

## 一、项目基本信息
- 应用类型：xxx
- 目标市场：xxx
- 技术栈概述：xxx
```

## 六、重要原则与提醒

### 6.1 本技能的边界

- ⚠️ **本技能提供的审查意见仅供参考，不构成正式法律意见。** 对于高风险项目（如涉及金融、医疗、大额交易），强烈建议用户咨询持证律师。
- ⚠️ **法规具有时效性和地域性。** 本技能基于截至2025年的主要法规，用户应关注最新立法动态。
- ⚠️ **AI生成内容的版权归属在不同司法管辖区认定不一。** 中国目前倾向于认为AI生成内容不构成作品（无人类独创性），但实践中存在争议。

### 6.2 给一人开发者的效率建议

| 痛点 | 低成本解决方案 |
|------|---------------|
| 素材版权 | 优先使用CC0/Unsplash免版税素材；商业项目购买Envato Elements订阅 |
| 字体授权 | 使用Google Fonts（全免费商用）或阿里巴巴普惠体 |
| 开源合规 | 从项目第一天起用`license-checker`扫描；避免引入GPL/AGPL库 |
| 隐私政策 | 使用iubenda/Termly等隐私政策生成器作为起点（需人工审核） |
| 商标查询 | 中国：商标局官网免费查询；美国：USPTO TESS；国际：WIPO Global Brand |
| AI生成内容 | 保留AI生成过程的记录（prompt、参数）；在输出中添加免责声明 |
